# 실시간 점수와 반복 확정 점수 UX 정리 스펙

**작성일:** 2026-04-22  
**상태:** Draft  
**관련 문서:**

---

## 1. 목적

운동 세션 화면에서 현재 하나의 숫자로 보이는 점수가 실제로는 다음 두 종류의 값을 번갈아 표시하고 있는 문제를 정리한다.

- 실시간 프레임 기반 자세 점수
- 반복 완료 후 확정되는 rep 점수

이번 스펙의 목적은 이 둘을 사용자 경험 관점에서 어떻게 다룰지 결정하고, 이후 구현 작업의 기준선을 만드는 것이다.

이번 문서는 설계와 계획만 다룬다. 런타임 로직 변경은 포함하지 않는다.

---

## 2. 배경 문제

현재 세션 화면의 메인 점수 영역은 운동 중과 반복 완료 후에 서로 다른 의미의 값을 표시한다.

### 2.1 현재 동작

- 운동 중에는 프레임 단위 점수의 누적값을 표시한다.
- rep가 완료되면 rep summary 기반 최종 점수로 값이 즉시 바뀐다.

즉, UI는 하나의 숫자를 보여주지만 내부적으로는 다른 두 계산 계층을 사용한다.

### 2.2 사용자 관점의 문제

- 사용자는 같은 숫자가 계속 이어진다고 기대한다.
- 실제로는 rep 종료 순간 점수가 갑자기 바뀌므로, 채점이 불안정하거나 신뢰하기 어렵다고 느낀다.
- 특히 스쿼트처럼 `depth_not_reached`, `lockout_incomplete` 같은 rep-level 판단이 들어가는 운동에서는 점프 폭이 커진다.

### 2.3 제품 관점의 문제

- 실시간 점수와 결과 점수의 목적이 다름에도 UI에서 구분되지 않는다.
- “왜 갑자기 55점이 됐는가” 같은 질문을 반복적으로 유발한다.
- 실시간 피드백 숫자가 오히려 결과 채점의 신뢰도를 떨어뜨린다.

---

## 3. 현재 구조 요약

### 3.1 실시간 점수

- 프레임별 포즈를 `scoringEngine.calculate()`로 평가한다.
- 이 값은 현재 자세 교정과 live breakdown 표시를 위한 값이다.
- rep 진행 중 UI는 `repCounter.getCurrentRepScore()`를 통해 이 누적 평균에 가까운 수치를 표시한다.

### 3.2 반복 확정 점수

- rep 종료 시 `repCounter.completeRep()`에서 rep summary를 만든다.
- 이후 `scoringEngine.scoreRep()`가 운동별 규칙으로 최종 rep 점수를 다시 계산한다.
- 이 값은 depth 도달, lockout 완료, confidence cap 등 rep 전체 품질을 반영한 결과 점수다.

### 3.3 핵심 해석

현재 구조는 다음과 같이 목적이 분리되어 있다.

- 실시간 점수: 즉시 교정용
- rep 점수: 결과 채점용

문제는 계산이 분리되어 있다는 사실이 아니라, 이를 UI에서 하나의 숫자로 합쳐 보여준다는 점이다.

---

## 4. 요구사항

이번 UX 정리는 다음 요구사항을 만족해야 한다.

1. 사용자가 “현재 보는 숫자의 의미”를 즉시 이해할 수 있어야 한다.
2. rep 종료 시 점수가 급변하더라도 “버그”처럼 느껴지지 않아야 한다.
3. 실시간 교정 가치는 유지하거나, 제거하더라도 더 나은 대체 UI가 있어야 한다.
4. 운동별 rep-level hard fail 구조는 유지할 수 있어야 한다.
5. 점수 체계 설명이 결과 페이지와 세션 페이지에서 충돌하지 않아야 한다.

---

## 5. 검토한 옵션

### 옵션 A. 현재 구조 유지

#### 설명

- 현재처럼 하나의 숫자 영역에서 실시간 점수와 rep 확정 점수를 모두 표시한다.

#### 장점

- 구현 변경이 없다.
- 사용자가 숫자를 계속 볼 수 있다.

#### 단점

- 의미가 섞인다.
- rep 종료 순간 점프가 계속 남는다.
- 사용자 신뢰도를 떨어뜨린다.

#### 결론

- 채택하지 않는다.

### 옵션 B. 실시간 점수와 rep 점수를 동시에 별도 표시

#### 설명

- `실시간 자세 점수`
- `최근 반복 점수`

를 별도 영역으로 분리한다.

#### 장점

- 의미가 명확하다.
- 내부 구조와 UI가 일치한다.

#### 단점

- 숫자 UI가 두 개가 된다.
- 운동 중 인지 부하가 커질 수 있다.
- 모바일 화면에서 정보 밀도가 높아진다.

#### 결론

- 가능하지만 우선순위는 높지 않다.

### 옵션 C. 메인 숫자는 rep 확정 점수만 유지하고, 실시간은 상태형 피드백으로 축소

#### 설명

- 메인 점수 숫자는 rep 완료 후에만 갱신한다.
- 운동 중에는 숫자 대신 다음 같은 상태형 피드백만 표시한다.
  - `좋음`
  - `주의`
  - `교정 필요`
- 상세 교정은 기존 breakdown / alert / visual overlay로 유지한다.

#### 장점

- 가장 이해하기 쉽다.
- rep 종료 후 점수 급변이 사라진다.
- 실시간 교정 기능은 유지할 수 있다.

#### 단점

- 숫자를 계속 보고 싶어하는 사용자는 아쉬울 수 있다.
- 기존 점수 UI 사용 습관이 바뀐다.

#### 결론

- 이번 스펙의 권장안으로 채택한다.

## 6. 최종 결정

최종 권장안은 **옵션 C**다.

### 6.1 제품 원칙

- 세션의 메인 숫자는 `반복 확정 점수`만 나타낸다.
- 운동 중에는 숫자형 실시간 점수를 메인 영역에 노출하지 않는다.
- 실시간 상태는 색상/텍스트/간단한 배지로 표현한다.

### 6.2 왜 이 방향인가

- rep 점수는 결과로 저장되고, 사용자도 결과 점수로 인식한다.
- 실시간 수치는 본질적으로 “중간 상태”이며 결과 점수와 동일한 의미를 갖지 않는다.
- 따라서 결과 의미가 강한 UI 슬롯에는 결과 점수만 두는 편이 맞다.

### 6.3 유지할 요소

- 실시간 breakdown 축적
- alert 기반 교정 문구
- skeleton/overlay 시각 피드백
- rep 종료 후 toast 또는 summary 메시지

### 6.4 제거 또는 축소할 요소

- 진행 중 메인 숫자 점수
- rep 도중 값이 오르내리는 live numeric display

## 7. 목표 UX

### 7.1 rep 시작 전

- 메인 점수: `--`
- 보조 문구: `rep 완료 후 점수가 표시됩니다`

### 7.2 rep 진행 중

- 메인 점수: 직전 rep 점수 유지 또는 `--`
- 실시간 상태 배지:
  - `좋음`
  - `주의`
  - `교정 필요`
- breakdown 영역:
  - 실시간으로 문제 metric만 요약 표시 가능

### 7.3 rep 완료 직후

- 메인 점수: 방금 완료한 rep의 확정 점수로 업데이트
- 피드백: rep-level 최종 문구 표시
- breakdown: 해당 rep의 최종 metric summary 표시

## 8. 범위

### 포함

- 세션 페이지 점수 표시 UX
- 세션 페이지 보조 텍스트/상태 배지
- 실시간 상태를 숫자 대신 표현하는 방식 정의
- rep 완료 후 점수 표시 기준 명확화

### 제외

- rep scoring 알고리즘 변경
- DB 저장 구조 변경
- 결과 페이지 점수 체계 변경
- 운동별 metric rule 변경
- 모바일 전용 레이아웃 리디자인

---

## 9. 구현 원칙

향후 구현 시 다음 원칙을 지킨다.

1. `scoreRep()`의 rep-level 최종 점수는 저장/표시 기준 점수로 유지한다.
2. `calculate()` 기반 실시간 수치는 메인 점수 숫자에서 분리한다.
3. 실시간 상태는 threshold 기반 상태 label로 변환한다.
4. 기존 alert / breakdown 로직은 가능한 한 재사용한다.
5. 세션 종료 export와 결과 페이지가 참조하는 최종 점수 의미는 바꾸지 않는다.

---

## 10. 위험 요소

### 10.1 숫자 피드백 감소에 대한 체감 저하

- 일부 사용자는 실시간 숫자 자체를 동기부여 요소로 볼 수 있다.
- 따라서 완전 제거 대신 `상태 배지 + 색상 + 교정 문구` 조합으로 대체해야 한다.

### 10.2 기존 UI 로직 결합도

- 현재 `updateScoreDisplay()`가 진행 중과 완료 후 상태를 모두 다룬다.
- 구현 시 조건 분기만 늘리면 코드가 더 복잡해질 수 있다.
- 따라서 점수 표시 책임과 실시간 상태 표시 책임을 나누는 정리가 필요하다.

---

## 11. 수용 기준

향후 구현 완료 시 다음을 만족해야 한다.

1. rep 진행 중 메인 점수 숫자가 급격히 튀지 않는다.
2. rep 완료 후에만 확정 점수가 메인 숫자 영역에 반영된다.
3. 사용자는 실시간 상태와 최종 점수를 혼동하지 않는다.
4. 기존 rep 저장/결과 페이지 점수 의미는 유지된다.
5. 세션 페이지 테스트에서 진행 중 상태와 완료 후 상태를 구분해 검증할 수 있다.
